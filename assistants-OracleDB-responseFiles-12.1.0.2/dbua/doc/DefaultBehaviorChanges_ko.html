Oracle Database 11g: 기본 기능의 변경 사항

이 페이지에서는 이전 릴리스에서 변경된 Oracle Database 11g의 일부 기능에 대해 설명합니다. 일부 매개변수의 기본값이 변경되거나 현재 스크립트나 응용 프로그램에 영향을 줄 수 있는 새로운 기능이나 요구 사항이 도입되었습니다. 11g에서는 여러 데이터베이스 관리 기능이 개선되었습니다. 자세한 내용은 Oracle Database New Features Guide 11g Release 1 (11.1)을 참조하십시오.

기능

설명
유니코드 5.0 지원
  • AL32UTF8 및 AL16UTF16 문자 집합에 대한 NLS 데이터 파일이 유니코드 표준 문자 데이터베이스 버전 5.0과 일치하도록 업데이트되었습니다. 이러한 개선 사항을 통해 Oracle 데이터베이스는 가장 최신 표준 버전을 지원합니다.
OracleJVM Java/JDK 5.0 호환성
  • OracleJVM은 Sun의 JDK 1.5를 지원합니다. 이 기능을 통해 J2SE 응용 프로그램(독립형 JDBC 또는 중간 계층)을 Oracle 데이터베이스로 이식하고 Java Annotations에서 지원하는 선언적 프로그래밍 스타일과 같은 J2SE 5.0의 장점을 활용할 수 있습니다.
OracleJVM JIT 구성
  • JIT가 기본적으로 사용으로 설정되며(즉시 사용 가능) 추가 구성이 필요하지 않습니다.
Linux와 Windows 간에 전송 가능한 데이터베이스
  • 이 기능을 사용하면 Linux와 Windows 간에 데이터베이스를 간단히 이동할 수 있습니다. 따라서 Linux와 Windows 간에 전송 가능한 데이터베이스를 만들 수 있으며, 이는 플랫폼 간 물리적 대기를 지원합니다.
손상된 블록 자동 보고
  • 인스턴스 복구 시 손상된 블록이 발견되면 DBA_CORRUPTION_LIST가 자동으로 채워집니다. 모든 백업, 매체 복구 및 인스턴스 복구 수준에서 블록 검증이 수행됩니다.
온라인 응용 프로그램 유지 관리 및 업그레이드

  • Oracle Database 11g에서는 Oracle 데이터베이스 응용 프로그램에 대한 패치와 업그레이드를 기록하고 적용할 때 함께 사용할 경우 Oracle Database 10g에 비해 작동 중지 시간을 현저히 줄일 수 있는 여러 기능이 도입되었습니다.

  • WAIT 옵션을 사용하는 DDL은 새 DEFAULT입니다. 대기 시간은 인스턴스 범위(초기화 매개변수 파일)에서 지정되고 세션 레벨에서 수정될 수 있습니다. WAIT 옵션으로 지정된 DDL 명령은 즉시 오류를 발생하는 대신 명령이 성공하도록 유예 기간을 정의하는 유연성을 제공합니다.

  • DEFAULT 값과 NOT NULL 제약 조건을 가진 새 열을 추가하는 ADD COLUMN 기능으로 더 이상 모든 기존 레코드에 기본값을 저장할 필요가 없습니다. 이 기능은 기존 데이터 볼륨과 관계없이 몇 초 만에 스키마 수정이 가능할 뿐만 아니라 공간을 소모하지도 않습니다.

  • 이전 릴리스에서는 메타 데이터가 전체 객체의 세분성을 가진 객체 간의 상호 종속성을 기록했습니다. 즉, 가끔 해당 작업을 수행하기 위한 논리적 요구 사항이 없는 경우 종속 객체가 무효화되었습니다. Oracle Database 11g는 종속성 메타 데이터를 더욱 미세한 세분성 레벨로 기록합니다. 종속된 객체의 변경 사항에 대한 응답에서 수반되는 종속성 객체의 무효화를 줄임으로써 응용 프로그램 가용성이 증가합니다.

  • 표시되지 않는 인덱스는 인덱스를 사용할 수 없게 만들거나 삭제하기 위한 대안입니다. 표시되지 않는 인덱스는 응용 프로그램의 특정 작업이나 모듈이 전체 응용 프로그램에 영향을 주지 않고 임시 인덱스 구조를 활용할 수 있게 합니다. 인덱스를 바로 삭제하지 않고 제거 테스트에 사용될 수 있으므로 운용 환경에서 테스트를 위한 유예 기간이 가능합니다.

  • 이번 Oracle 데이터베이스 릴리스에서는 구체화된 뷰 로그에 대한 세션 레벨 제어 기능이 추가되었습니다. 개별 세션에 대해 구체화된 뷰에 대한 변경 사항의 캡처(구체화된 뷰 로그)를 사용 안함으로 설정하고 다른 세션에 의해 수행된 변경 사항은 계속 기록할 수 있습니다. 이 기능은 응용 프로그램 업그레이드에 필요한 작동 중지 시간을 줄여줍니다.

  • 이전 릴리스에서는 테이블에 로그나 구체화된 뷰가 정의된 경우 해당 테이블을 재정의할 수 없었습니다. 그러나 이번 릴리스에서는 트리거, 인덱스 및 기타 유사한 종속 객체와 마찬가지로 구체화된 뷰 로그를 중간으로 복제할 수 있습니다. 재정의 종료 시 ROWID 로그가 무효화됩니다. 처음에 종속된 모든 구체화된 뷰가 전체 새로 고침을 수행해야 합니다.

  • [온라인 테이블 재정의 후 종속 PL/SQL 재컴파일 최소화] 기능은 온라인 테이블 재정의 이후 종속 PL/SQL 패키지의 재컴파일 필요성을 최소화합니다. 이 기능은 온라인 테이블 재정의 이후 종속 PL/SQL을 수동으로 재컴파일하기 위한 시간과 노력을 줄입니다. 또한 논리적으로 재정의의 영향을 받지 않는 뷰, 동의어 및 기타 테이블 종속 객체(트리거는 예외)를 포함합니다.

  • 개선된 온라인 인덱스 생성 및 재구축 기능은 온라인 인덱스 생성 또는 재구축 시 배타적 잠금의 필요성을 없앰으로써 이러한 작업을 완전히 투명하게 만듭니다.

  • Oracle Database 11g에서 새 ALTER TABLE 구문이 도입되었습니다. 운영 체제는 해당 소유자에 대해서도 파일을 읽기 전용으로 만들도록 선례를 설정합니다. 이전에는 해당 테이블의 소유자가 아닌 사용자에 대해서만 테이블을 읽기 전용으로 만들 수 있었습니다.
가상 열

  • 가상 열은 표현식의 평가를 통해 정의되며 그 결과는 테이블에 대한 열의 메타 데이터가 됩니다. 가상 열은 테이블 생성 또는 수정 시 정의될 수 있습니다.

  • 가상 열은 응용 프로그램 개발자가 공간을 소모하지 않고 계산 및 변환을 테이블의 열(메타 데이터) 정의로 정의할 수 있도록 합니다. 따라서 응용 프로그램 개발 시 이러한 가상 열에 대한 최적기에 추가 통계를 제공함으로써 오류를 줄이고 질의 최적화를 개선할 수 있습니다.
Oracle XML DB에 대해 이진 XML 지원

  • 이진 XML(XML 문서 구조를 나타내는 방법)은 데이터베이스에서 유지되는 XML에 대해 세번째 저장 영역 모델을 추가합니다. 이진 XML은 기존 객체 저장 영역 및 텍스트 저장 영역 모델을 대체하지 않고 보완합니다.

  • XML 스키마의 사용 가능 여부에 관계없이 XML 작업을 상당히 최적화할 수 있고 XML이 있는 위치가 디스크이거나 메모리이거나 또는 네트워크이거나 관계없이 XML의 내부 표현을 동일하게 유지할 수 있습니다.
XMLIndex
  • 이 기능을 사용하면 조직에서 인덱스화할 XML의 형태나 콘텐츠에 대한 사전 지식이 없어도 XML 문서를 효율적으로 인덱스화할 수 있습니다. 매우 효율적인 인덱스화된 XML 콘텐츠의 XPath 기반 검색 기능을 완벽하게 지원합니다.
자동 진단 저장소

  • ADR(자동 진단 저장소)는 추적 파일 및 기타 오류 진단 데이터를 저장하고 구성하기 위한 새 시스템 관리 저장소입니다. ADR은 데이터베이스에서 발생한 모든 심각한 오류를 종합적으로 볼 수 있으며 문제 진단 및 해결에 필요한 모든 관련 데이터를 유지합니다.

  • 모든 데이터베이스 진단 정보의 저장, 형식 지정 및 검색을 위한 정형화되고 일관된 메커니즘을 제공합니다. ADR은 심각한 오류에 대한 사고를 자동으로 생성하고 사고 관리 기능을 제공함으로써 고객의 문제 해결 시간을 상당히 줄여줍니다.
자동 상태 모니터링

  • 자동 상태 모니터링은 사전에 데이터베이스의 상태를 검사하고 데이터베이스에 영향을 줄 수 있는 문제가 있는지 확인합니다. DBA는 자동 상태 모니터링의 도움을 받아 현재 데이터베이스의 상태를 종합적으로 살펴보고 데이터베이스에 영향을 주는 문제를 해결하는 방법을 찾을 수 있습니다.

  • 구성 요소나 상황에 따라 자동 상태 모니터링은 잘못된 구성 요소나 리소스를 격리하거나 문제를 해결합니다. 이 기능을 통해 다운 시간을 최소화하고 서비스 중단에 앞서 계획을 수립할 수 있습니다.
ASM 롤링 업그레이드
  • 롤링 업그레이드 기능을 사용하면 데이터베이스의 가용성에 영향을 주지 않고 독립적인 ASM 클러스터의 노드를 이전 또는 패치할 수 있습니다. 롤링 업그레이드 기능은 더 많은 작동 시간을 제공하며 새 릴리스로 무리없이 이전할 수 있습니다.
ASM 관리를 위한 새 SYSASM 롤
  • 이 기능은 데이터베이스 관리 책임과 저장 영역 관리 책임을 분리할 수 있는 새 SYSASM 권한을 제공합니다. SYSASM 권한을 사용하면 관리자가 여러 데이터베이스에서 공유되는 디스크 그룹을 관리할 수 있습니다. SYSASM 권한은 SYSDBA 권한의 의무와 분명히 구분됩니다.
단일 인스턴스 ASM을 클러스터화된 ASM으로 변환

  • 이 기능은 모든 모드에서 ASM을 암시적으로 구성함으로써 Enterprise Manager 내에서 클러스터화되지 않은 ASM 데이터베이스를 클러스터화된 ASM 데이터베이스로 변환하는 기능을 지원합니다. 또한 대기 데이터베이스를 지원하기 위해 단일 인스턴스를 Oracle RAC 변환 유틸리티로 확장합니다.

  • 변환을 단순화함으로써 고객이 더 쉽게 데이터베이스를 이전하고 확장성 및 고가용성과 같은 Oracle RAC에서 제공하는 이점을 활용할 수 있습니다.
데이터베이스 재생
  • 데이터베이스 재생 기능은 사용자가 운용 시스템에서 실제 데이터베이스 작업 로드를 캡처하고 테스트 시스템에서 재생하는 방식으로 실제와 같은 테스트를 수행할 수 있도록 합니다. 또한 잠재적인 문제(예: 오류 발생 및 성능 저하)와 이러한 문제를 해결할 수 있는 권장 방법을 제시하는 분석 및 보고 기능을 제공합니다.
SQL 재생
  • SQL 재생은 변경 전후의 응답 횟수를 비교하고 대조하여 SQL 문의 성능에 미치는 변경의 영향을 손쉽게 평가할 수 있는 방법을 제공합니다. SQL 재생을 사용하면 운용 데이터베이스와 같은 소스 시스템에서 SQL 작업 로드를 캡처하고 변경 사항이 적용된 테스트 시스템에서 재생할 수 있습니다.
데이터베이스 복제 개선 사항
  • 이 기능은 소스 및 대상 스테이지 영역에 대한 요구 사항을 없애고, 일반 RMAN 백업 설정에서 복제를 지원하고, Enterprise Manager 데이터베이스 복제 페이지에서 작업 요약 사용자 인터페이스를 개선합니다.
개선된 적응 측정 단위 임계값

  • 적응 측정 단위 임계값 기능은 데이터베이스 성능 측정 단위에 대한 경보 임계값의 선택을 단순화하고 개선합니다. 이 기능은 DBA가 최소한의 노력으로 고품질 기준 요소 구동 성능 경보 임계값을 구성할 수 있도록 지원합니다.

  • AWR 기준 요소와의 완벽한 통합은 측정 단위 통계를 수집하는 원천입니다.

  • 신속한 구성 옵션은 OLTP 또는 데이터 웨어하우스 시스템 프로파일을 기반으로 원클릭 기본 경보 임계값 집합을 제공합니다.

  • SYSTEM 변동 기간 기준 요소에 대한 시간 그룹 선택이 자동화되었습니다.

  • 알려진 문제에 대한 응답 레벨을 기반으로 후보 경보 측정 단위 검색을 유도했습니다.

  • 측정 단위 차트 작성 및 분석이 개선되었습니다.
자동 작업
  • Oracle Database 11g에서 자동 작업은 관리성 창에서 수백 또는 수천 개의 작업을 실행할 수 있습니다. 이러한 고관리성 작업 로드를 자동으로 관리하기 위해 필요한 스케줄러 기반 구조를 추가합니다.
자동 유지 관리 작업 관리
  • 이 기능은 자동 최적기 통계 수집, 자동 세그먼트 권고자 등과 같은 다양한 데이터베이스 유지 관리 작업 간에 즉시 사용 가능한 리소스 분배(CPU 및 I/O) 관리 기능을 제공합니다. CPU는 자동으로 관리됩니다. I/O는 I/O 리소스 관리자가 사용으로 설정된 경우에만 관리됩니다.
자동 작업 로드 저장소(AWR) 기준 요소

  • 자동 작업 로드 저장소(AWR) 기준 요소는 특정 기간을 미래의 특정 시점에 비교될 수 있는 관심있는 기간으로 지정함으로써 정확한 성능 비교를 수행할 수 있습니다.

  • 이 강력한 기능은 동적인 미래의 기준 요소를 정의할 수 있는 기능을 제공함으로써 비교에 사용되는 성능 데이터의 생성 및 관리 프로세스를 단순화합니다.
DBMS_JOB을 Oracle Scheduler로 이전
  • DBMS_JOB 사용자가 작업을 Oracle Scheduler로 이전할 수 있습니다. 이 경우 DBMS_JOB 작업이 Oracle Scheduler 작업이 되고 Oracle Scheduler의 모든 기능을 활용할 수 있습니다. DBMS_JOB 사용자는 Oracle Scheduler 기능을 활용할 수 있습니다.
단순화된 초기화 매개변수 관리
  • 이 릴리스는 더욱 확대된 SPFILE 결함 허용 범위, 손쉬운 SPFILE의 손실 복구, 서버 매개변수 파일 및 초기화 매개변수 값을 더 쉽게 관리할 수 있는 더욱 직관적인 Enterprise Manager 초기화 매개변수 관리 인터페이스와 같은 개선된 기능을 제공합니다.
개선된 OFA(Optimal Flexible Architecture) 지원

  • 데이터베이스 설치 프로세스가 ORACLE_BASE 환경 변수를 기반으로 하도록 재설계되었습니다. 지금까지 이 변수 설정은 선택 사항이었으며 유일한 필수 변수는 ORACLE_HOME이었습니다. 이 기능에서 ORACLE_BASE는 유일한 필수 입력이고 ORACLE_HOME 설정은 ORACLE_BASE에서 파생됩니다. 이러한 변경은 Oracle 소프트웨어 설치 구성 방법을 합리화하고 따라서 진행 중인 관리를 더 쉽게 만듭니다.

  • 이 기능은 기본 Oracle 데이터베이스 설치에서 OFA(Optimal Flexible Architecture) 사양을 더욱 잘 준수하도록 만들어 관리성을 개선합니다.
관리성 관련 개선 사항

  • Automatic Storage Management의 이 새로운 기능은 저장 영역 관리 자동화를 확장하고, 확장성을 개선하고, Oracle 데이터베이스 파일에 대한 관리를 단순화합니다.

  • 이전 릴리스에서 Advanced Replication은 트리거 기반 복제 방법이었습니다. Oracle Database 11 g에서 이 기능은 기존 기능을 웹 기반 Enterprise Manager Database Control로 변환합니다. 이 기능은 11 g Enterprise Manager Database Control에서 Advanced Replication을 관리하는 방식으로 기존 Advanced Replication 고객을 지원합니다.

  • ADDM은 종합적인 클러스터 범위 성능 진단 및 튜닝 권고를 제공하도록 개선되었습니다. ADDM의 특수 모드는 Oracle RAC 데이터베이스를 분석하고 전체 클러스터에 영향을 주는 문제와 개별 인스턴스에 영향을 주는 문제에 대해 보고합니다.

    이 기능은 I/O 및 상호 접속 트래픽과 같은 전역 리소스의 튜닝을 지원하고 Oracle RAC 데이터베이스를 쉽고 자세히 튜닝할 수 있도록 합니다.

  • Oracle 데이터베이스는 현재 로드가 많은 SQL 문을 자동으로 감지한 다음 해당 SQL 프로파일을 생성함으로써 유지 관리 창에서 자동으로 튜닝합니다. 또한 로드가 많은 SQL 문의 성능을 현저히 개선하는 인덱스와 같은 새 액세스 구조를 생성하도록 사전 권고를 제시합니다.

  • Streams 프로세스가 사용 안함으로 설정되면 자동으로 경보가 생성됩니다. 또한 주요 Streams 측정 단위에 대해 사용자 정의된 임계값을 기반으로 경보가 생성됩니다. 이러한 경보는 Streams 관리자에게 핵심 Streams 구성 요소에 대한 정보를 제공합니다.
메모리 관련 개선 사항

  • 자동 메모리 관리 인스턴스는 메모리 크기(SGA 및 PGA)를 튜닝하고 필요에 따라 SGA(시스템 전역 영역)과 PGA 합계(합계 프로그램 전역 영역) 간에 메모리를 재분배합니다. 메모리 크기(SGA 및 PGA) 초기화 매개변수는 동적이며 데이터베이스를 다시 시작하지 않고 언제든지 변경할 수 있습니다. 자동 메모리 관리 옵션이 선택되면 데이터베이스 인스턴스가 SGA와 PGA 크기를 자동으로 관리합니다.

  • 이전에는 PL/SQL에 의해 캐시되는 SQL 커서 수가 OPEN_CURSORS에 의해 결정되었습니다. 11g에서는 캐시되는 커서 수가 SESSION_CACHED_CURSORS에 의해 결정됩니다. Oracle Database Reference 설명서를 참조하십시오.

  • 공유 풀 오버헤드에 필요한 공간이 포함되도록 SHARED_POOL_SIZE를 늘려야 합니다.

  • DB_BLOCK_SIZE의 기본값은 운영 체제에 따라 다르지만 일반적으로 8KB입니다. (이전 릴리스에서는 일반적으로 2KB였습니다.)
트랜잭션 및 공간

  • 이제 AUM(자동 실행 취소 관리) 기본 설정이 기본적으로 사용으로 설정됩니다. PL/SQL 프로시저는 각 개별 환경에 대해 실행 취소 테이블스페이스의 크기를 알맞게 조정합니다. 새 릴리스로 업그레이드된 데이터베이스는 AUM으로 원활히 이전할 수 있습니다.

  • 분할된 객체에 대한 통계 수집은 분할된 객체에 대한 통계 수집의 속도와 정확성을 획기적으로 향상시키는 개선된 통계 수집 프로세스입니다.

  • 단순화된 임시 공간 관리는 DBA가 사용 중인 임시 공간의 크기와 해당 공간을 사용하는 사람 및 더 많은 공간이 필요한지 여부를 쉽게 파악할 수 있도록 함으로써 임시 공간 관리 문제를 해결합니다. 이 기능은 잘못된 임시 공간 구성으로 인한 오류를 줄이고 분실된 질의와 같은 응용 프로그램 디자인 문제를 식별합니다.
Oracle Base 및 진단 대상 구성

  • ORACLE_BASE에 대해 DBCA "인터뷰 단계" 동안 지정하는 디렉토리는 Oracle 홈 인벤토리에 DIAGNOSTIC_DEST 매개변수로 저장되고 후속 디렉토리 생성 시 사용됩니다 (기본값은 $ORACLE_BASE 디렉토리). 또한 ADD INSTANCE 작업 동안 DBUA는 ORACLE_BASE 값이 모든 노드에서 동일한 것으로 간주하고 모든 노드에서 해당 값을 검증합니다.

  • 진단 대상 위치에는 모든 ADR 디렉토리(경보 로그 등과 같은 진단 파일)가 포함됩니다. 이 진단 대상 디렉토리는 11g 이전의 Oracle 데이터베이스를 11g 이상의 데이터베이스 릴리스로 업그레이드하는 데 필요합니다. Oracle Base 디렉토리가 이미 존재하면 Upgrade Assistant가 자동으로 이 정보를 검색하고 해당 경로를 미리 채웁니다.

  • Oracle Database 릴리스 11g부터는 백그라운드 덤프, 사용자 덤프 및 코어 덤프 대상에 대한 초기화 매개변수 설정이 진단 대상으로 바뀌었습니다.
XE 업그레이드

  • 단일 인스턴스 데이터베이스의 경우 DBUA(Database Configuration Upgrade Assistant) 구성 유틸리티를 사용하여 Oracle Database Express Edition(Oracle Database XE)에서 Oracle Database 11g로 업그레이드할 수 있습니다. DBUA는 해당 데이터베이스가 XE 데이터베이스인지 여부를 식별합니다.

  • XE 데이터베이스 파일은 ORACLE_BASE/oradata/XE 경로 아래 있습니다. 사용자가 업그레이드 후에 XE 홈을 제거할 수도 있기 때문에 이 파일을 새 위치로 복사해야 합니다.
데이터 파일을 ASM, SAN 및 기타 파일 시스템으로 이동

  • 업그레이드 시 ASM이나 OFS 또는 SAN(Storage Area Networks) 및 NAS(Network Area Storage)와 같은 기타 저장 영역 장치로 데이터 파일을 이동할 수 있습니다. 업그레이드 시 데이터베이스 파일을 이동하는 경우 디스크의 밸런싱을 조정하고 SAN, NAS 또는 ASM과 같은 더 효율적인 저장 영역 장치로 파일을 이동함으로써 이 테이블스페이스에 대한 기본 작동 중지 시간을 활용할 수 있습니다.
업그레이드/다운그레이드

  • 데이터베이스가 업그레이드된 원본 버전을 기반으로 주요 릴리스 및 패치 집합 릴리스를 모두 다운그레이드할 수 있습니다. 주요 릴리스 다운그레이드는 10.2 및 10.1에 대해 지원됩니다. 패치 집합 다운그레이드는 11.1 이전의 모든 패치 릴리스에 대해 지원됩니다.
    Oracle Database Express Edition.W에서 업그레이드된 데이터베이스는 다운그레이드할 수 없습니다.
    업그레이드한 릴리스로만 다운그레이드할 수 있습니다. 예를 들어, 10.1에서 11.1로 업그레이드한 경우 10.2로 다운그레이드할 수 없고 10.1로만 다운그레이드할 수 있습니다.

  • Oracle10g Release 1(10.1) 데이터베이스의 릴리스 번호가 10.1.0.5보다 낮은 경우 다운그레이드에 앞서 10.1 릴리스의 최신 패치를 설치해야 합니다. 마찬가지로 Oracle 10g Release 2(10.2) 데이터베이스의 릴리스 번호가 10.2.0.3보다 낮은 경우 다운그레이드에 앞서 10.2 릴리스의 최신 패치를 설치해야 합니다. 10.1 또는 10.2의 패치 릴리스에서 업그레이드할 수 있지만 ORACLE_HOME에 다운그레이드 후에 사용할 최신 패치 릴리스를 설치해야 합니다.

  • 데이터베이스의 메시지 게이트웨이나 작업 영역 관리자는 모두 10.1.0.6 및 10.2.0.4 릴리스 이전의 Oracle 데이터베이스 패치 집합의 일부가 아닙니다. 따라서 다운그레이드하기 전에 10.2.0.3 또는 10.1.0.5 ORACLE_HOME에 모든 관련 패치를 별도로 적용해야 합니다.

  • 최소 로그 파일 및 기본 로그 파일의 크기가 증가되었습니다. 기본이 100MB인 OMF(Oracle Managed Files)를 사용하지 않는 한 최소 로그 파일 크기는 이제 4MB이고 기본값은 50MB입니다.

자동 확장 시스템 파일에 대한 명령행 옵션

  • AUTOEXTEND 명령행 옵션은 업그레이드의 일부로 데이터 파일의 자동 확장을 사용합니다.이 옵션은 업그레이드 시 데이터 파일을 자동으로 확장하고 업그레이드 후에 자동 확장을 원래의 설정으로 되돌립니다. 이 옵션은 디스크 공간이 충분하고 새 데이터 파일을 추가하거나 파일 크기를 수동으로 늘릴 필요가 없는 경우에 유용합니다.
